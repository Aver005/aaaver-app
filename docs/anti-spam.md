# Анти-спам: как работает форма

Идея: не мучить людей картинками со светофорами, а сделать отправку
дорогой для ботов и незаметной для человека. Защита собрана из нескольких
слоёв — каждый по отдельности обходится, вместе они отсекают практически
весь автоматический спам.

## Слои

### 1. Proof-of-work (жёсткий)

При первом касании формы клиент запрашивает `POST /api/challenge`
и получает `{ id, salt, difficulty }`. Дальше в Web Worker'е подбирается
`nonce`, при котором `sha256(salt + ":" + nonce)` начинается
с `difficulty` нулевых бит.

- При сложности 14 это ~16 тысяч хешей — доли секунды даже на телефоне.
- Решение начинается, пока человек печатает, поэтому к моменту отправки
  оно уже готово и никто ничего не ждёт.
- Challenge одноразовый (гасится атомарным UPDATE) и живёт 15 минут.
- Для спамера это значит: каждое письмо стоит CPU-времени, а решения
  нельзя накопить впрок.

Сложность настраивается через `POW_DIFFICULTY` (env). Каждый +1 бит —
вдвое дороже. 14 — комфортно, 16 — заметно злее, выше 18 уже может
раздражать владельцев старых телефонов.

### 2. Тайминг (жёсткий)

Сервер отклоняет отправку, если с момента выдачи challenge прошло меньше
2 секунд (`minSubmitDelayMs` в `server/config.ts`). Человек физически
не успеет заполнить три поля быстрее, скрипт — успеет.

### 3. Rate-limit по IP (жёсткий)

`RATE_LIMIT_PER_HOUR` (5) и `RATE_LIMIT_PER_DAY` (20) на отправку,
плюс 30 challenge'ей в час. События живут в таблице `attempts`.
Если сервер стоит за nginx/caddy — поставьте `TRUST_PROXY=1`,
иначе все посетители будут «одним IP» и лимиты сработают на всех сразу.

### 4. Honeypot (тихий)

В форме есть скрытое поле `website`. Люди его не видят, тупые боты
заполняют. Такое сообщение сервер принимает с ответом «ок» (чтобы не
подсказывать боту, что его раскусили), помечает `delivered = -1`
и никуда не отправляет. В БД остаётся для разбора.

### 5. Slide-to-send (мягкий)

Кнопки «отправить» нет — есть ползунок. Клиент собирает статистику жеста
(длительность, число сэмплов, пройденное расстояние) и шлёт её вместе
с формой. Сервер ничего не блокирует по жесту (иначе сломаем клавиатурную
доступность — Enter на ползунке тоже отправляет), но пишет флаги
в `spam_flags`:

- `gesture-weak` — жест был, но подозрительно короткий/ровный;
- `gesture-keyboard` — жеста не было (клавиатура или скрипт).

Флаги видны в Telegram-сообщении и в БД — по ним легко отличить
сомнительные письма.

## Что происходит при отправке

```
клиент                              сервер
──────────────────────────────────────────────────────────────
focus формы → POST /api/challenge → создаёт challenge (uuid, salt)
решает pow в воркере (фоном)
…человек печатает…
тянет ползунок →
POST /api/contact ────────────────→ rate-limit по IP
                                    challenge: есть? не использован? жив?
                                    pow верный?
                                    прошло ≥ 2с с выдачи?
                                    honeypot пуст?
                                    длины полей в норме?
                                    → INSERT в messages
                                    → sendMessage в Telegram
                                    ← 200 {ok}
```

Если Telegram недоступен или chat_id ещё не известен — сообщение лежит
в SQLite с `delivered = 0`, фоновая задача дошлёт его позже (раз в 10
минут, до 20 попыток).

## Как покрутить

Всё в `.env`:

```
POW_DIFFICULTY=14        # битов нулей; +1 = вдвое дороже для бота
RATE_LIMIT_PER_HOUR=5
RATE_LIMIT_PER_DAY=20
TRUST_PROXY=0            # 1 — если за реверс-прокси
```

Посмотреть, что прилетало, включая спам:

```sh
sqlite3 data/aaaver.db "SELECT id, name, delivered, spam_flags, datetime(created_at,'unixepoch') FROM messages ORDER BY id DESC LIMIT 20;"
```
